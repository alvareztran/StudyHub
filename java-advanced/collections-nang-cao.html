16.1 ArrayList vs LinkedList

Khái niệm và mục đích sử dụng

Cả ArrayList và LinkedList đều kế thừa từ interface List. Chúng có đặc tính chung là cho phép lưu giữ các phần tử trùng lặp và duy trì đúng thứ tự thêm vào. Tuy nhiên, cấu trúc tổ chức bộ nhớ vật lý của chúng hoàn toàn khác nhau.

Cơ chế hoạt động bên trong

ArrayList (Mạng động): Sử dụng một mảng thô (Array) tĩnh nội bộ để lưu trữ. Khi mảng bị đầy phần tử, Java tự động cấp phát một mảng mới có kích thước nhân bản lên gấp **1.5 lần** mảng cũ, tiến hành copy toàn bộ các phần tử sang mảng mới và hủy mảng cũ đi.

LinkedList (Danh sách liên kết đôi): Sử dụng cấu trúc các node liên kết trỏ chéo lẫn nhau. Mỗi Node vật lý trên Heap chứa dữ liệu và hai con trỏ prev (phần tử đứng trước) và next (phần tử đứng sau):

null ◄─── [ Node A ] ───► ◄─── [ Node B ] ───► ◄─── [ Node C ] ───► null
Bảng so sánh độ phức tạp thuật toán
Thao tác ArrayList (Dynamic Array) LinkedList (Doubly Linked List)
Truy xuất theo chỉ số: get(index)$O(1)$ (Cực nhanh nhờ công thức tính địa chỉ trực tiếp)$O(N)$ (Chậm do phải duyệt tuần tự từ đầu hoặc cuối danh sách)
Thêm phần tử vào cuối: add(value)$O(1)$ (Trừ trường hợp mảng bị đầy cần tái cấu trúc mảng mới)$O(1)$ (Chỉ cập nhật con trỏ Node cuối)
Thêm/Xóa phần tử ở đầu: add(0, value)$O(N)$ (Phải dịch chuyển toàn bộ các phần tử phía sau sang phải 1 đơn vị)$O(1)$ (Chỉ cần thay đổi con trỏ đầu)
Xóa ở vị trí bất kỳ: remove(index)$O(N)$ (Phải dịch các phần tử lấp khoảng trống)$O(1)$ (Nếu đã định vị được Node cần xóa, chỉ cần đổi con trỏ)

Kết luận thực tế: Trừ phi bạn cần liên tục thực hiện thêm/xóa phần tử ở đầu danh sách, trong **90% dự án thực tế** ta luôn ưu tiên lựa chọn **ArrayList** nhờ tốc độ đọc vượt trội.

16.2 HashSet vs TreeSet

Khái niệm và mục đích sử dụng

Cả HashSet và TreeSet đều triển khai từ interface Set, với đặc tính cốt lõi là **không cho phép các phần tử trùng lặp**.

Cơ chế hoạt động bên trong

HashSet: Hoạt động dựa trên một bảng băm **HashMap** chạy ngầm bên dưới. Mỗi phần tử add vào HashSet thực chất là lưu vào vị trí Key của HashMap với giá trị Value là một hằng số dummy Object dùng chung.

TreeSet: Hoạt động dựa trên **TreeMap**, sử dụng cấu trúc cây tự cân bằng **Cây Đỏ Đen (Red-Black Tree)** để tự động sắp xếp các phần tử theo thứ tự tăng dần sau mỗi lần thêm mới.

So sánh kỹ thuật
Tiêu chí HashSet TreeSet
Thứ tự phần tửKhông đảm bảo thứ tự thêm vào.Tự động sắp xếp tăng dần.

16.3 HashMap Internal (Cơ chế hoạt động bên trong HashMap)

Bản chất lưu trữ dữ liệu

HashMap là cấu trúc dạng bảng băm lưu trữ các cặp dữ liệu dạng Key-Value. Vùng nhớ lưu trữ của HashMap là một mảng các ô Bucket (mặc định ban đầu có dung lượng 16 buckets).

Cơ chế ghi nhận phần tử: put(key, value)
  1. JVM gọi hàm key.hashCode() để lấy mã băm của khóa.
  2. HashMap lấy mã băm này thực hiện thuật toán modulo kích thước mảng để tính toán ra vị trí **Bucket Index** cần lưu trữ.
  3. Nếu tại ô bucket chưa có phần tử nào: HashMap lưu trữ phần tử đó vào ô bucket ngay lập tức ($O(1)$).
  4. Xử lý va chạm băm (Collision): Nếu ô bucket đã bị chiếm dụng bởi key khác, HashMap gọi phương thức equals() để so sánh nội dung hai key:
    • Nếu equals() trả về true: Ghi đè (Update) đè giá trị cũ bằng value mới.
    • Nếu equals() trả về false: Gắn thêm phần tử mới vào cuối cấu trúc danh sách liên kết **Linked List** tại ô bucket đó.
Sự cải tiến vượt bậc trong Java 8 (Treeify chuyển đổi sang cây đỏ đen)

Ở Java 7 trở về trước, va chạm băm dài sẽ biến bucket thành 1 danh sách liên kết đơn dài, làm giảm hiệu năng tìm kiếm từ $O(1)$ xuống $O(N)$ trong trường hợp xấu nhất.

Từ Java 8+, nếu số lượng phần tử va chạm tại một bucket vượt quá ngưỡng **8 phần tử (TREEIFY_THRESHOLD)**, Java lập tức tự động chuyển đổi danh sách liên kết đơn này thành cấu trúc **Cây Đỏ Đen (Red-Black Tree)**. Giúp tối ưu hóa tốc độ tìm kiếm tại bucket va chạm từ **$O(N)$** xuống còn **$O(\log N)$**.

Va chạm băm tại Bucket 5: Java 7: [Bucket 5] ──> Node A ──> Node B ──> Node C ... (Danh sách liên kết O(N)) Java 8+: [Bucket 5] ──> Node B (Root) ├── Node A (Left) └── Node C (Right) (Cây đỏ đen tự cân bằng O(log N))

16.4 TreeMap Internal

Khái niệm và cơ chế hoạt động

TreeMap là bản đồ tự động sắp xếp các phần tử theo thứ tự tăng dần của các Khóa (Keys). Nó hoạt động dựa trên cấu trúc dữ liệu **Cây đỏ đen tự cân bằng (Self-Balancing Binary Search Tree)**.

Mỗi khi thêm (put) hoặc xóa (remove) một phần tử, TreeMap thực hiện quay các node của cây và đổi màu sắc đỏ/đen để đảm bảo cây luôn đạt trạng thái cân bằng tối ưu, giúp độ phức tạp của các thao tác cơ bản luôn giữ ở mức **$O(\log N)$**.

16.5 & 16.6 Sắp xếp dữ liệu với Comparable và Comparator

1. Comparable (Quy tắc sắp xếp nội bộ mặc định)

Dùng khi lớp thực thể của bạn muốn có một quy tắc sắp xếp ngầm định mặc định duy nhất. Để sử dụng, Class bắt buộc implement interface Comparable<T> và override phương thức compareTo():

class Student implements Comparable<Student> {
    private int age;
    public Student(int age) { this.age = age; }

    @Override
    public int compareTo(Student other) {
        return this.age - other.age; // Sắp xếp theo tuổi tăng dần
    }
}
// Sử dụng:
Collections.sort(studentsList); // Tự động sắp xếp theo tuổi
2. Comparator (Quy tắc sắp xếp linh hoạt từ bên ngoài)

Dùng khi bạn muốn định nghĩa nhiều tiêu chí sắp xếp khác nhau mà không cần sửa đổi mã nguồn gốc của Class, hoặc để cấu hình bộ lọc động. Ta sử dụng phương thức compare():

// Định nghĩa bộ so sánh theo Tên
Comparator<Student> byName = (s1, s2) -> s1.getName().compareTo(s2.getName());

// Định nghĩa bộ so sánh theo Tuổi đảo ngược (giảm dần)
Comparator<Student> byAgeDesc = (s1, s2) -> Integer.compare(s2.getAge(), s1.getAge());

// Sử dụng thực tế:
studentsList.sort(byName);
studentsList.sort(byAgeDesc);
