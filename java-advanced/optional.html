Tổng quan về lớp Optional

Khái niệm và sự ra đời

Lớp java.util.Optional<T> được giới thiệu từ **Java 8** là một lớp wrapper đối tượng chứa giá trị có thể tồn tại hoặc rỗng (null). Mục đích tối thượng của Optional là giải quyết tận gốc lỗi sập ứng dụng kinh điển **`NullPointerException` (NPE)**.

Sự cải tiến so với kiểm tra null truyền thống:

// Trước Java 8 (Dễ bị quên kiểm tra null dẫn đến NPE):
String city = null;
if (user != null && user.getAddress() != null) {
    city = user.getAddress().getCity();
}

// Từ Java 8 (Viết mạch lạc an toàn tuyệt đối):
String city = Optional.ofNullable(user)
                      .map(User::getAddress)
                      .map(Address::getCity)
                      .orElse("Default City");

20.1 Optional.of() & 20.2 Optional.ofNullable()

1. Optional.of(value) - Khởi tạo chắc chắn không Null

Dùng để tạo một đối tượng Optional bao bọc lấy giá trị truyền vào khi bạn đảm bảo giá trị đó **chắc chắn không bao giờ rỗng**.

⚠️ Cảnh báo: Nếu bạn truyền giá trị null vào phương thức Optional.of(null), JVM sẽ lập tức ném ra lỗi NullPointerException ngay tại dòng khởi tạo. 

Optional<String> appName = Optional.of("My Application"); // OK
Optional<String> appNameNull = Optional.of(null); // Crash app lập tức!
2. Optional.ofNullable(value) - Khởi tạo cho phép Null

Dùng để bao bọc một giá trị có thể có dữ liệu hoặc có thể bị rỗng. Nếu giá trị truyền vào là null, phương thức tự động trả về một đối tượng **`Optional.empty()`** thay vì ném lỗi.

String name = null;
Optional<String> opt = Optional.ofNullable(name); // Trả về Optional.empty() an toàn

20.3 orElse() vs orElseGet() & 20.4 ifPresent()

1. orElse(defaultValue) vs orElseGet(Supplier)

Cả hai đều dùng để định nghĩa giá trị mặc định thay thế nếu Optional ở trạng thái rỗng. Tuy nhiên, có sự khác biệt rất lớn về mặt hiệu năng chạy chương trình:

  • orElse(defaultValue): Giá trị mặc định truyền vào **luôn luôn được tính toán/khởi tạo** bất kể Optional có chứa dữ liệu thực hay không.
  • orElseGet(Supplier): Sử dụng lazy-evaluation, giá trị mặc định **chỉ được khởi tạo khi và chỉ khi Optional thực sự rỗng** (tiết kiệm tài nguyên RAM/CPU).
// orElse (Gây lãng phí hiệu năng vì databaseQuery() luôn chạy):
String name = Optional.of("Bang").orElse(databaseQuery()); 

// orElseGet (Tối ưu: databaseQuery() chỉ chạy khi name rỗng):
String name = Optional.of("Bang").orElseGet(() -> databaseQuery());
2. ifPresent(Consumer) - Thực thi hành động khi có dữ liệu

Nếu Optional chứa dữ liệu, thực thi khối code Lambda bên trong. Nếu rỗng, bỏ qua an toàn không ném lỗi.

Optional<String> opt = Optional.of("Java");
opt.ifPresent(System.out::println); // In ra "Java"

Quy chuẩn sử dụng Optional trong Spring Boot (JPA)

Trong các dự án Backend hiện đại, Optional là chuẩn mực trả về của tầng Database Access (Spring Data JPA) để ép lập trình viên phải xử lý trường hợp không tìm thấy bản ghi:

// Kiểu viết cũ (Dễ sót kiểm tra null):
User user = userRepository.findById(id);
if (user == null) {
    throw new ResourceNotFoundException("User not found");
}

// Kiểu viết chuẩn hóa hiện đại kết hợp Lambda:
User user = userRepository.findById(id)
                          .orElseThrow(() -> new ResourceNotFoundException("User not found"));
Quy tắc vàng khi đi làm (Best Practices)
  • KHÔNG dùng Optional làm tham số truyền vào của phương thức: public void save(Optional<User> user) (Bị coi là bad practice). Hãy truyền đối tượng thô.
  • KHÔNG dùng Optional làm thuộc tính (field) của Class/Entity: Gây khó khăn cho quá trình Serialization dữ liệu. Chỉ nên dùng làm kiểu dữ liệu trả về của phương thức.
  • TRÁNH gọi hàm get() trực tiếp: Gọi optional.get() mà không kiểm tra isPresent() trước đó sẽ phát sinh lỗi NoSuchElementException nếu Optional rỗng. Hãy luôn ưu tiên dùng orElse, orElseGet hoặc orElseThrow.
