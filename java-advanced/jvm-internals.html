Tổng quan Máy ảo Java (JVM)

Khái niệm và vai trò

JVM (Java Virtual Machine - Máy ảo Java) là môi trường thực thi ảo hóa bytecode của Java, chịu trách nhiệm chuyển dịch file trung gian .class thành mã máy (Machine Code) tương thích với hệ điều hành và CPU vật lý bên dưới. Đây là nền tảng của khẩu hiệu huyền thoại **"Write Once, Run Anywhere"**.

[ Java Code .java ] ──( javac )──> [ Bytecode .class ] ──( Class Loader ) │ ▼ (Quét RAM) [ CPU thực thi ] ◄── Machine Code ◄── Execution Engine ◄── Runtime Data Areas

27.1 Class Loader Subsystem (Hệ thống nạp Class)

Vòng đời nạp Class vào bộ nhớ

Quá trình Class Loader tải tệp bytecode lên RAM diễn ra qua 3 giai đoạn độc lập:

  1. Loading (Nạp tệp): Đọc file vật lý .class từ ổ đĩa và ánh xạ thành cấu trúc nhị phân java.lang.Class lưu trữ ở vùng nhớ Metaspace (Method Area).
  2. Linking (Liên kết):
    • Verification: Kiểm tra tính hợp lệ của bytecode, đảm bảo an ninh hệ thống (không chọc phá ô nhớ bất hợp pháp).
    • Preparation: Cấp phát vùng nhớ cho các biến static và gán giá trị mặc định ban đầu (ví dụ: static int được gán bằng 0).
    • Resolution: Chuyển đổi Symbolic References (tham chiếu ký hiệu bằng chuỗi) trong bytecode thành Direct References (địa chỉ ô nhớ vật lý trực tiếp).
  3. Initialization (Khởi tạo): Chạy khối mã static block static { ... } và gán giá trị thực tế cho các biến static (ví dụ gán static int từ 0 thành giá trị cấu hình thực tế).
Mô hình phân quyền ủy thác (Parent Delegation Model)

JVM tổ chức Class Loader phân cấp theo hình cây. Khi nhận yêu cầu nạp Class, Loader hiện tại luôn ủy quyền kiểm tra lên Loader cha trước, nếu cha không tìm thấy thì con mới tự quét nạp:

[ Bootstrap ClassLoader ] (Nạp gói lõi jdk: java.lang.*, java.util.*) ▲ (Ủy quyền ngược lên) [ Platform ClassLoader ] (Nạp các gói mở rộng extension) ▲ (Ủy quyền ngược lên) [ Application ClassLoader ] (Nạp class trong source code của lập trình viên)

Lợi ích: Ngăn chặn việc ghi đè bất hợp pháp lên các class hệ thống (ví dụ: hacker cố tình định nghĩa class java.lang.String giả mạo).

27.2 Garbage Collection (Minor GC vs Major GC)

Cơ chế phát hiện rác - Reachability Analysis

Java sử dụng cơ chế dọn rác tự động thông qua bộ **Garbage Collector (GC)**. Để xác định đối tượng nào đã chết, GC dùng thuật toán phân tích khả năng tiếp cận (Reachability Analysis) xuất phát từ các điểm gốc **GC Roots** (Biến cục bộ Stack, luồng đang chạy Active Thread, biến tĩnh Static). Nếu đối tượng không có đường đi kết nối tới GC Roots, nó lập tức bị coi là rác.

Phân chia vùng nhớ trong Java Heap
┌───────────────────────────────── HEAP MEMORY ──────────────────────────────────┐ │ Young Generation (Chứa đối tượng ngắn hạn) │ Old Generation │ │ ┌───────────────────┬──────────────┬────────────────┐ │ (Đối tượng sống lâu) │ │ │ Eden Space │ Survivor S0 │ Survivor S1 │ │ │ └─┴───────────────────┴──────────────┴────────────────┴─┴───────────────────────┘
Sự khác biệt giữa Minor GC và Major GC
Tiêu chí so sánh Minor GC (Young Gen GC) Major GC (Old Gen GC)
Phân vùng xử lý Chỉ quét dọn vùng Young Generation (Eden, S0, S1). Quét dọn vùng Old Generation.
Cơ chế thăng hạng Object mới sinh nằm ở Eden. Sau mỗi lần Minor GC còn sống sót sẽ tăng tuổi, chuyển qua S0/S1. Object vượt qua ngưỡng tuổi quy định (mặc định là 15 tuổi) sẽ thăng hạng lên Old Gen.
Tần suất & Pause Time Xử lý rất thường xuyên, tốc độ dọn dẹp cực nhanh (Pause time nhỏ). Ít xảy ra hơn, thời gian xử lý rất lâu và có thể làm đóng băng app (Stop The World).

Full GC: Là chu trình dọn dẹp toàn diện dọn sạch cả Young Gen, Old Gen và Metaspace. Đây là tác vụ đắt đỏ nhất và có nguy cơ gây trễ (latency) nghiêm trọng cho ứng dụng Backend.

27.5 Java Memory Model (JMM)

Tính năng và vai trò của JMM

Java Memory Model định nghĩa các quy tắc kiểm soát luồng đọc/ghi dữ liệu của các Thread giữa bộ nhớ làm việc riêng của CPU (Working Memory) và bộ nhớ chính của RAM (Main Memory).

Khái niệm Happens-Before Relationship

Happens-before là tập quy tắc của JMM nhằm đảm bảo rằng các thao tác ghi dữ liệu của Thread A tại một thời điểm chắc chắn sẽ hiển thị thấy được (visible) đối với thao tác đọc dữ liệu của Thread B sau đó. Ví dụ: Thao tác mở khóa `unlock` một Monitor Lock luôn xảy ra trước (`happens-before`) thao tác `lock` tiếp theo trên cùng một lock.

Các tiêu chuẩn cốt lõi
  • Atomicity (Tính nguyên tử): Đảm bảo tác vụ chạy trọn vẹn không bị ngắt quãng giữa chừng (ví dụ dùng block synchronized).
  • Visibility (Tính hiển thị): Đảm bảo thay đổi giá trị của Thread A hiển thị tức thì sang Thread B (ví dụ dùng biến volatile).
  • Ordering (Tính thứ tự): Ngăn chặn JVM và CPU tự ý sắp xếp lại thứ tự thực hiện câu lệnh (Instruction Reordering) làm biến đổi logic của mã nguồn đa luồng.
