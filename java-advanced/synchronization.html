Đồng bộ hóa luồng (Synchronization)

Khái niệm và sự cần thiết

Khi ứng dụng vận hành đa luồng, nhiều Thread có thể đồng thời truy cập và ghi dữ liệu vào một tài nguyên dùng chung (Shared Resource). Nếu không có cơ chế điều phối đồng bộ, dữ liệu sẽ bị sai lệch ngoài kiểm soát.

Synchronization cung cấp giải pháp bảo vệ dữ liệu bằng cách ép buộc các Thread phải xếp hàng truy cập tài nguyên tuần tự tại các vùng mã được bảo vệ.

25.1 Race Condition (Tranh chấp tài nguyên)

Khái niệm và cơ chế Lost Update

Race Condition xảy ra khi kết quả cuối cùng của chương trình phụ thuộc vào thứ tự thực thi ngẫu nhiên của các luồng. Lỗi phổ biến nhất là Lost Update (Mất lượt cập nhật).

Ví dụ: Thao tác tăng số count++ trông như một dòng lệnh nguyên tử, nhưng thực chất JVM biên dịch nó thành 3 bước Bytecode độc lập:

Bước 1: Đọc count từ RAM (ví dụ: count = 5) Bước 2: Tăng giá trị lên 1 ở CPU Register (5 + 1 = 6) Bước 3: Ghi lại giá trị mới vào ô nhớ RAM

Nếu hai Thread cùng đọc giá trị 5 cùng lúc, cả hai sẽ tính toán ra 6 và ghi đè 6 vào RAM. Lượt tăng của một Thread đã bị nuốt mất.

// Đoạn mã tạo lỗi Race Condition
class Counter {
    int count = 0;
    public void increment() { count++; } // Thiếu đồng bộ hóa!
}

25.2 Từ khóa synchronized (Khóa giám sát Monitor Lock)

Cơ chế Monitor Lock

Mỗi Object trong Java đều sở hữu một khóa nội bộ ẩn gọi là Monitor Lock (hoặc Intrinsic Lock). Khi một Thread đi vào vùng mã được đánh dấu synchronized, nó lập tức chiếm giữ khóa này. Mọi Thread khác muốn truy cập đều bị block và xếp vào hàng đợi cho đến khi Thread đầu tiên hoàn thành và nhả khóa.

Các hình thức đồng bộ hóa
  • Synchronized Method (Đồng bộ hóa phương thức): Khóa đối tượng gọi hàm (this): 
    public synchronized void increment() { count++; }
  • Synchronized Block (Đồng bộ hóa khối lệnh): Tối ưu hiệu năng bằng cách chỉ khóa đoạn mã thực sự ghi dữ liệu: 
    public void increment() {
    // Chỉ khóa khối mã tính toán, cho phép chạy song song đoạn mã khác
    synchronized(this) { count++; }
}
  • Synchronized Static Method: Khóa đối tượng Class Loader (Counter.class) của JVM thay vì instance: 
    public static synchronized void resetAll() { totalCount = 0; }

25.3 Từ khóa volatile (Đảm bảo Visibility bộ nhớ)

Khái niệm và cơ chế hoạt động

Trong kiến trúc máy tính, mỗi nhân CPU có cache riêng (L1/L2/L3). Khi một luồng chạy, nó lưu bản sao của biến vào CPU Cache của nhân đó để tăng tốc truy cập. Điều này làm phát sinh lỗi **Memory Visibility (Tính hiển thị bộ nhớ)**: Luồng A thay đổi giá trị của biến ở nhân 1 nhưng luồng B ở nhân 2 vẫn đọc giá trị cũ lưu trong cache nhân 2.

Đánh dấu biến bằng từ khóa `volatile` ép buộc JVM phải luôn đọc và ghi biến trực tiếp từ **Main Memory (Bộ nhớ chính RAM)**, đảm bảo mọi luồng luôn thấy giá trị mới nhất.

private volatile boolean running = true; // Luôn đọc ghi trực tiếp từ RAM

⚠️ Lưu ý cực kỳ quan trọng: volatile KHÔNG giải quyết được Race Condition (không đảm bảo Atomicity). Do đó volatile int count = 0; count++; vẫn bị sai lệch giá trị khi chạy đa luồng.

25.4 Deadlock (Khóa chết)

Khái niệm và kịch bản xảy ra

Deadlock là hiện tượng treo hệ thống vĩnh viễn khi từ hai luồng trở lên bị kẹt vào vòng lặp chờ khóa lẫn nhau vô hạn.

[ Thread A ] ── giữ Lock 1 ── muốn lấy Lock 2 ──┐ ▲ │ (Chờ nhau vô hạn) └────────── muốn lấy Lock 1 ── giữ Lock 2 ──[ Thread B ]
4 Điều kiện kích hoạt Deadlock (Coffman Conditions)
  1. Mutual Exclusion: Tài nguyên không thể chia sẻ (chỉ 1 luồng giữ lock).
  2. Hold and Wait: Luồng đang giữ lock này và đòi xin thêm lock khác.
  3. No Preemption: Không thể cưỡng đoạt lock từ luồng khác.
  4. Circular Wait: Chờ đợi xoay vòng (A chờ B, B chờ A).
Cách phòng tránh Deadlock
  • Sắp xếp thứ tự lấy khóa đồng nhất: Ép mọi luồng phải lấy Lock 1 trước rồi mới lấy Lock 2.
  • Sử dụng ReentrantLock: Dùng lock.tryLock(timeout) để tự động nhả khóa nếu đợi quá lâu, không chờ vô hạn.

So sánh volatile vs synchronized

Tiêu chí so sánh Từ khóa volatile Từ khóa synchronized
Đảm bảo VisibilityCó (Đọc ghi thẳng từ RAM)Có (Đồng bộ cache khi nhả lock)
Đảm bảo tính nguyên tử (Atomicity)Không (Hàm như count++ vẫn bị lỗi)Có (Ngăn chặn Race Condition hoàn toàn)
Gây nghẽn luồng (Blocking)Không (Non-blocking, hiệu năng cao)Có (Luồng khác phải xếp hàng chờ đợi lock)
