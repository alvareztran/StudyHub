14.1 Stack Memory (Vùng nhớ Stack)

Khái niệm và mục đích sử dụng

Stack Memory (Bộ nhớ ngăn xếp) là vùng nhớ trong JVM chịu trách nhiệm lưu trữ thông tin tạm thời phục vụ lời gọi hàm và luồng thực thi của chương trình. Bộ nhớ Stack bao gồm:

  • Các biến cục bộ (Local Variables) thuộc kiểu nguyên thủy.
  • Các biến tham chiếu (Reference Variables) chứa địa chỉ đối tượng.
  • Thông tin tham số truyền vào của phương thức.

Cơ chế phân tách luồng: Mỗi Thread (luồng chạy) trong Java đều sở hữu riêng biệt một vùng nhớ Stack độc lập hoàn toàn, không chia sẻ cho các luồng khác.

Cơ chế hoạt động bên trong (LIFO - Last In First Out)

Stack hoạt động chặt chẽ theo cơ chế vào sau ra trước. Mỗi khi một phương thức được gọi, JVM lập tức tạo mới một Stack Frame đẩy (push) lên đỉnh của Stack. Khi phương thức chạy xong, Stack Frame đó bị hủy (pop) lập tức.

Hãy phân tích ví dụ gọi hàm lồng nhau:

public class Main {
    public static void main(String[] args) {
        methodA();
    }
    public static void methodA() {
        methodB();
    }
    public static void methodB() {
        int number = 10;
    }
}

Quá trình nạp xếp chồng Stack Frame tương ứng trong JVM:

Khi methodB() đang chạy: Khi methodB() chạy xong: ┌─────────────────────────┐ ┌─────────────────────────┐ │ Stack Frame: methodB() │ │ │ (Đã pop giải phóng) ├─────────────────────────┤ ├─────────────────────────┤ │ Stack Frame: methodA() │ │ Stack Frame: methodA() │ ├─────────────────────────┤ ├─────────────────────────┤ │ Stack Frame: main() │ │ Stack Frame: main() │ └─────────────────────────┘ └─────────────────────────┘
Cú pháp và quy tắc

Tất cả biến khai báo cục bộ bên trong hàm đều được lưu trữ trực tiếp trên Stack:

int age = 20; // Giá trị 20 nằm trực tiếp tại Stack Frame

Biến đối tượng: Biến tham chiếu nằm ở Stack nhưng giá trị của nó chỉ là địa chỉ pointer trỏ sang Heap:

User user = new User(); // Biến "user" nằm ở Stack, đối tượng thực tế "User" nằm ở Heap
Ví dụ thực tế trong dự án

Trong kiến trúc web Spring Boot, mỗi Request gửi tới API được xử lý bởi một Thread độc lập riêng biệt từ Thread Pool. Nhờ mỗi Thread có Stack riêng, các biến cục bộ khai báo trong các phương thức của Service hoàn toàn cô lập, an toàn đa luồng (Thread-safe) mà không cần khóa đồng bộ.

14.2 Heap Memory (Vùng nhớ Heap)

Khái niệm và mục đích sử dụng

Heap Memory là vùng nhớ lớn nhất trong JVM, được chia sẻ dùng chung cho toàn bộ tất cả mọi Threads trong chương trình. Nó được sử dụng để chứa:

  • Toàn bộ các thực thể đối tượng (Objects) được khởi tạo bằng từ khóa new.
  • Các mảng dữ liệu (Arrays).
  • Các Collection Framework (ArrayList, HashMap...).
  • Các String Objects tạo qua new.
Cơ chế hoạt động bên trong

Ví dụ: User user = new User();

Vùng nhớ STACK Vùng nhớ HEAP ┌───────────────┐ ┌───────────────────────────────┐ │ user: 0x5501 ├────────────────>│ Address 0x5501: User Object │ └───────────────┘ │ - name = null, age = 0 │ └───────────────────────────────┘

Khi ta ngắt liên kết tham chiếu bằng cách gán: user = null;, đối tượng tại địa chỉ 0x5501 không còn bất kỳ biến tham chiếu nào quản lý nữa. Nó trở thành **Garbage Object (Đối tượng rác)** và sẽ bị bộ dọn rác Garbage Collector quét qua thu hồi bộ nhớ vào chu kỳ kế tiếp.

Ví dụ mã nguồn đơn giản
User user1 = new User();
User user2 = user1; // u1 và u2 cùng trỏ về 1 đối tượng ở Heap

user1 = null; // Ngắt u1, đối tượng vẫn tồn tại trên Heap vì còn u2 trỏ tới

14.3 String Pool (Hồ chứa chuỗi)

Khái niệm và mục đích sử dụng

String Pool là một phân vùng bộ nhớ đặc biệt nằm trong Heap dùng để lưu trữ các chuỗi String Literal để tái sử dụng tối ưu RAM.

Cơ chế phân bổ bộ nhớ khi khai báo String Literal vs New String:

Khai báo s1 = "Java", s2 = "Java": Khai báo s3 = new String("Java"): ┌──────────────┐ ┌──────────────┐ │ s1: 0x101 ├──┐ │ s3: 0x909 ├─> [Heap Object 0x909] ├──────────────┤ │ └──────────────┘ │ │ s2: 0x101 ├──┼─> [String Pool] ▼ └──────────────┘ └──> "Java" (0x101) "Java" (String Pool)

Phép so sánh địa chỉ:

  • s1 == s2 trả về true (cùng tham chiếu String Pool).
  • s1 == s3 trả về false (s3 trỏ qua đối tượng Heap trung gian).

14.4 Method Area (Vùng nhớ Phương thức)

Khái niệm và mục đích sử dụng

Method Area là vùng nhớ dùng để lưu giữ thông tin cấp Class (Class-level data) phục vụ cấu trúc hoạt động chung của chương trình. Vùng nhớ này được dùng chung cho tất cả các luồng (Threads), bao gồm:

  • Cấu trúc thông tin Metadata của lớp (Class Metadata).
  • Mã Bytecode thô của tất cả phương thức.
  • Các biến tĩnh (Static Variables) dùng chung của Class.
  • Runtime Constant Pool chứa các hằng số.
Cơ chế hoạt động bên trong

Vì biến static được lưu tại Method Area, chúng có vòng đời tồn tại độc lập với sự sống của đối tượng. Ta có thể gọi trực tiếp thông qua tên lớp mà không cần sử dụng từ khóa new để tạo đối tượng.

class Counter {
    static int count = 0;
    Counter() { count++; }
}

// Thực thi:
new Counter();
new Counter();
System.out.println(Counter.count); // Kết quả in ra: 2 (Dùng chung ô nhớ static duy nhất)

14.5 Metaspace

Khái niệm và mục đích sử dụng

Metaspace là vùng nhớ chứa Class Metadata được Java giới thiệu từ phiên bản **Java 8 trở lên** nhằm thay thế hoàn toàn cho phân vùng **Permanent Generation (PermGen)** cũ.

Bảng so sánh sự thay đổi kiến trúc giữa PermGen và Metaspace:

Tiêu chí so sánh PermGen (Java 7 trở xuống) Metaspace (Java 8 trở lên)
Vị trí bộ nhớ vật lýNằm bên trong phân vùng Heap JVM cấu hình cứng.Nằm trực tiếp trên **Bộ nhớ ngoài Native Memory** của hệ điều hành.
Giới hạn kích thước tối đaCố định (Fixed). Dễ dẫn đến lỗi tràn bộ nhớ `OutOfMemoryError: PermGen space`.Tự động co giãn theo dung lượng RAM trống của hệ điều hành.
Cách cấu hình tham số-XX:MaxPermSize=128m-XX:MaxMetaspaceSize=256m
Ví dụ thực tế trong dự án

Các ứng dụng Spring Boot lớn sử dụng rất nhiều framework như Hibernate, Lombok, Jackson để tự động sinh ra các class động (Dynamic Proxy Class) tại thời điểm Runtime. Metaspace giúp hệ thống tự động co giãn ô nhớ tải các class động này một cách an toàn mà không lo bị tràn bộ nhớ cứng như PermGen trước đây.

Tổng quan phân bổ bộ nhớ trong JVM

BỘ NHỚ JVM ┌─────────────────────────────────────────────────────────────┐ │ [ Vùng nhớ riêng của từng Thread (Thread-Private) ] │ │ ┌───────────────────────────────────────────────────────┐ │ │ │ Stack Memory (Lưu Stack Frames, Local vars, Pointer) │ │ │ └───────────────────────────────────────────────────────┘ │ ├─────────────────────────────────────────────────────────────┤ │ [ Vùng nhớ dùng chung cho tất cả Threads (Thread-Shared) ] │ │ ┌───────────────────────────────────────────────────────┐ │ │ │ Heap Memory (Lưu đối tượng Object, Array, Collection) │ │ │ ├───────────────────────────────────────────────────────┤ │ │ │ String Pool (Lưu trữ hằng chuỗi tái sử dụng) │ │ │ ├───────────────────────────────────────────────────────┤ │ │ │ Method Area (Lưu Class Metadata, static variables) │ │ │ └───────────────────────────────────────────────────────┘ │ └─────────────────────────────────────────────────────────────┘ *Lưu ý: Metaspace lưu Metadata của Class nằm ở Native Memory của OS.

Các lỗi bộ nhớ thường gặp & Phỏng vấn tuyển dụng

1. Lỗi tràn bộ nhớ Stack (java.lang.StackOverflowError)

Xảy ra khi vùng nhớ Stack bị đầy do số lượng Stack Frame vượt quá giới hạn thiết lập.

  • Nguyên nhân phổ biến: Lập trình thuật toán đệ quy vô hạn (không có điểm dừng) hoặc các phương thức gọi lặp lồng nhau quá sâu.
  • Mã ví dụ lỗi: 
    public void test() {
    test(); // Đệ quy liên tục làm sập Stack lập tức
}
2. Lỗi tràn bộ nhớ Heap (java.lang.OutOfMemoryError: Java heap space)

Xảy ra khi vùng nhớ Heap không còn đủ dung lượng trống để cấp phát bộ nhớ cho đối tượng mới và bộ dọn rác GC đã cố gắng hết sức nhưng không thể giải phóng thêm.

  • Nguyên nhân phổ biến: Tạo quá nhiều đối tượng có kích thước lớn, hoặc xảy ra lỗi **Rò rỉ bộ nhớ (Memory Leak)** do lưu giữ tham chiếu đối tượng rác trong các danh sách tĩnh static khiến GC chịu thua không thể xóa.
  • Mã ví dụ lỗi: 
    List<byte[]> list = new ArrayList<>();
while(true) {
    list.add(new byte[1024 * 1024]); // Nạp liên tục mảng 1MB vào Heap cho đến sập app
}
