Mẫu thiết kế phần mềm (Design Patterns)

Khái niệm và sự ra đời

Design Patterns (Mẫu thiết kế) là những giải pháp tối ưu, đã được chuẩn hóa và tái sử dụng nhiều lần để giải quyết các vấn đề thiết kế kiến trúc phần mềm phổ biến trong thực tế.

Design Pattern không phải là thư viện hay framework cụ thể. Nó là cẩm nang hướng dẫn cách tổ chức mối quan hệ, cấu trúc và tương tác giữa các Class/Object sao cho mã nguồn dễ mở rộng (extensible), dễ bảo trì (maintainable) và giảm thiểu sự phụ thuộc (loose coupling).

Phân loại 3 nhóm mẫu thiết kế lớn
  • Creational Patterns (Nhóm khởi tạo): Tập trung vào việc ẩn logic khởi tạo đối tượng phức tạp khỏi phía Client (ví dụ: Singleton, Factory, Builder).
  • Structural Patterns (Nhóm cấu trúc): Giải quyết cách kết hợp các class và đối tượng thành các cấu trúc lớn hơn (ví dụ: Decorator, Adapter, Proxy).
  • Behavioral Patterns (Nhóm hành vi): Giải quyết giao tiếp và phân bổ trách nhiệm giữa các đối tượng (ví dụ: Strategy, Observer).

28.1 Singleton, 28.2 Factory & 28.3 Builder

1. Singleton Pattern - Khởi tạo duy nhất 1 thực thể

Đảm bảo một Class chỉ có duy nhất một thực thể trong suốt vòng đời ứng dụng. Constructor bắt buộc phải đặt là private để chặn Client gọi new tự do:

public class DatabaseConfig {
    private static DatabaseConfig instance;
    private DatabaseConfig() {} // Private Constructor

    public static synchronized DatabaseConfig getInstance() {
        if (instance == null) {
            instance = new DatabaseConfig(); // Lazy Initialization
        }
        return instance;
    }
}

Ứng dụng: Spring Bean mặc định đều quản lý ở dạng Singleton Scope.

2. Factory Pattern - Hàm máy sản xuất đối tượng

Ẩn giấu hoàn toàn logic tạo lập đối tượng cụ thể (Concrete Class) đằng sau một Interface chung. Client chỉ cần truyền tham số yêu cầu và nhận về đối tượng tương ứng:

public class AnimalFactory {
    public static Animal create(String type) {
        if ("dog".equalsIgnoreCase(type)) return new Dog();
        if ("cat".equalsIgnoreCase(type)) return new Cat();
        throw new IllegalArgumentException("Unknown animal type");
    }
}

Ứng dụng: BeanFactory trong Spring Boot, SessionFactory trong Hibernate.

3. Builder Pattern - Tạo lập đối tượng phức tạp step-by-step

Giải quyết vấn đề Constructor có quá nhiều tham số (telescoping constructors) gây khó đọc và nhầm lẫn thứ tự kiểu dữ liệu. Cho phép thiết lập từng thuộc tính tùy chọn:

User user = new User.Builder()
                        .name("Bang")
                        .age(21)
                        .address("Can Tho")
                        .build();

Ứng dụng: Lombok annotation @Builder tự sinh code builder lúc biên dịch.

28.4 Strategy, 28.5 Observer & 28.6 Decorator

1. Strategy Pattern - Linh hoạt thay đổi thuật toán ở Runtime

Định nghĩa một họ thuật toán độc lập, đóng gói từng thuật toán và cho phép hoán đổi chúng linh hoạt lúc chạy chương trình mà không cần thay đổi logic Client:

// Thay đổi cổng thanh toán linh hoạt
PaymentService service = new PaymentService(new MomoPayment());
service.checkout(50000); // Thanh toán qua Momo

service.setStrategy(new PaypalPayment());
service.checkout(50000); // Đổi sang PayPal tức thì

Ứng dụng: Spring Security AuthenticationProvider.

3. Decorator Pattern - Trang trí mở rộng chức năng động

Cho phép thêm các tính năng, hành vi mới vào một đối tượng sẵn có ở thời điểm runtime mà không làm ảnh hưởng đến cấu trúc nguyên bản của Class gốc và không lạm dụng kế thừa:

// Xây dựng luồng đọc dữ liệu có bộ đệm (Buffered)
InputStream fileStream = new FileInputStream("data.txt");
InputStream bufferedStream = new BufferedInputStream(fileStream); // Decorator mở rộng tính năng buffer
