Tổng quan Concurrency Framework

Tại sao cần Concurrency Framework?

Tạo một luồng vật lý qua new Thread() là thao tác cực kỳ đắt đỏ vì JVM phải yêu cầu hệ điều hành cấp phát bộ nhớ Stack riêng. Nếu ứng dụng có hàng nghìn tác vụ đồng thời mà tạo luồng thủ công, máy chủ sẽ nhanh chóng cạn kiệt RAM và sập hệ thống (OutOfMemoryError).

Concurrency Framework ra đời nhằm tái sử dụng luồng hiệu quả (Thread Pool), phân tách luồng nghiệp vụ (Business logic) khỏi việc quản trị hạ tầng luồng.

26.1 ExecutorService & 26.2 Thread Pool

1. ExecutorService - Dịch vụ điều phối tác vụ

ExecutorService là trung tâm của Java Concurrency, chịu trách nhiệm nhận tác vụ từ hàng đợi (Blocking Queue) và phân bổ cho các Worker Thread xử lý.

  • execute(Runnable): Gửi tác vụ kiểu void, không thu hồi kết quả và không ném exception.
  • submit(Callable/Runnable): Trả về một đối tượng Future<T> cho phép kiểm soát kết quả sau khi luồng xử lý xong.
2. Thread Pool - Kiến trúc hàng đợi và Worker
[ Tác vụ gửi vào ] ──> [ Blocking Queue ] │ ▼ (Phân phối động) [ Thread Pool (Các Thread chạy sẵn) ] ├── Worker Thread 1 (Đang xử lý) ├── Worker Thread 2 (Rảnh rỗi) └── Worker Thread 3 (Rảnh rỗi)
Phân loại các loại Thread Pool trong Java
Phương thức khởi tạo Đặc điểm vận hành Ứng dụng thực tế thích hợp
Executors.newFixedThreadPool(n) Duy trì cố định $n$ Thread. Tác vụ dư thừa xếp hàng đợi trong Queue. Xử lý request với số tải biết trước, kiểm soát RAM ổn định.
Executors.newCachedThreadPool() Tự động tạo thread mới nếu bận, tự giải phóng thread nếu rảnh quá 60s. Nhiều tác vụ ngắn hạn, không phù hợp khi tải cao (dễ gây tràn luồng).
Executors.newSingleThreadExecutor() Chỉ duy trì đúng 1 Thread duy nhất. Tác vụ cần chạy tuần tự cách biệt để tránh Race Condition.
Executors.newScheduledThreadPool(n) Luồng cho phép lên lịch chạy định kỳ. Tác vụ định kỳ (Cron job, dọn dẹp cache, gửi email báo cáo hàng ngày).

26.3 CompletableFuture (Lập trình bất đồng bộ Non-blocking)

Hạn chế của Future truyền thống

Khi sử dụng Future.get(), luồng hiện tại sẽ bị khóa nghẽn (blocking) cho tới khi tác vụ nền hoàn thành. Điều này phá vỡ tính tối ưu xử lý bất đồng bộ.

CompletableFuture - Cơ chế Reactive Non-blocking

CompletableFuture cho phép viết code theo phong cách Reactive chuỗi (Chaining) thông qua các callback hàm. Luồng chính không bao giờ bị block:

import java.util.concurrent.CompletableFuture;

public class Main {
    public static void main(String[] args) throws Exception {
        CompletableFuture.supplyAsync(() -> "vietnam")
            .thenApply(String::toUpperCase) // Biến đổi kết quả: "VIETNAM"
            .thenAccept(result -> System.out.println("Kết quả in ra: " + result)) // Tiêu thụ kết quả
            .exceptionally(ex -> {
                System.out.println("Xảy ra lỗi: " + ex.getMessage());
                return null;
            });
        
        System.out.println("Main Thread tiếp tục chạy mượt mà không bị block!");
        Thread.sleep(1000); // Giữ chương trình để xem kết quả async
    }
}

Các phương thức liên kết nâng cao:

  • thenCompose(): Kết hợp chuỗi 2 tác vụ bất đồng bộ liên tiếp (tác vụ sau phụ thuộc kết quả tác vụ trước).
  • CompletableFuture.allOf(): Đợi tất cả các tác vụ chạy song song hoàn thành đồng loạt rồi mới xử lý tiếp.

26.4 Concurrent Collections (Tập hợp dữ liệu an toàn đa luồng)

Sự bất an của Collection thông thường

Các Class như ArrayList, HashMap hoàn toàn **không an toàn** khi nhiều luồng cùng thao tác ghi dữ liệu, gây lỗi hỏng cấu trúc dữ liệu hoặc vòng lặp vô hạn ở JVM.

Các cấu trúc Concurrent Collections chuyên dụng
  • `ConcurrentHashMap`: Thay thế hoàn toàn cho HashMap trong môi trường đa luồng. Nó sử dụng cơ chế chia nhỏ khóa (Striped Locking/Segment Lock) giúp các luồng có thể ghi dữ liệu đồng thời vào các phân vùng khác nhau của map mà không cần khóa toàn bộ đối tượng, mang lại hiệu năng tối đa.
  • `CopyOnWriteArrayList`: Mỗi lần thực hiện thao tác ghi (add, set), cấu trúc này sẽ sao chép toàn bộ mảng cũ sang một bản sao mảng mới để ghi. Nhờ đó các luồng đọc có thể đọc cực nhanh mà không cần khóa. Thích hợp cho các trường hợp **Đọc rất nhiều, Ghi rất ít** (ví dụ danh sách cấu hình hệ thống).
  • `BlockingQueue`: Hàng đợi tự động block luồng. Nếu hàng đợi rỗng, luồng đọc (Consumer) tự động bị dừng lại chờ cho đến khi luồng viết (Producer) đẩy dữ liệu vào. Đây là cấu trúc cốt lõi để triển khai mô hình Producer-Consumer hoặc Message Queue nội bộ.
