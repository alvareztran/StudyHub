8.1 Abstract Class (Lớp trừu tượng)

Khái niệm và mục đích sử dụng

Abstract Class là lớp được khai báo kèm từ khóa abstract. Lớp này không thể khởi tạo đối tượng trực tiếp qua từ khóa new mà chỉ dùng làm lớp cha cơ sở cho các lớp con kế thừa mở rộng.

Mục đích sử dụng:

  • Thiết lập bộ khung thuộc tính (fields) và hành vi chung cho các lớp con cùng dòng họ
  • Ép buộc các lớp con kế thừa bắt buộc phải định nghĩa lại các phương thức trừu tượng (abstract methods)
  • Hỗ trợ tối ưu hóa và tái sử dụng mã nguồn
Cơ chế hoạt động bên trong

Abstract Class có thể chứa cả các phương thức rỗng không có code thân (Abstract Method) và phương thức có mã nguồn đầy đủ (Concrete Method).

Nếu một Class con kế thừa từ Abstract Class:

  • Bắt buộc phải ghi đè (Override) toàn bộ các abstract methods của cha.
  • Hoặc nếu không ghi đè, bản thân lớp con đó cũng phải được khai báo là một abstract class.
Cú pháp và quy tắc
abstract class Animal {
    abstract void sound(); // Phương thức abstract không có thân hàm

    void sleep() {
        System.out.println("Sleeping..."); // Phương thức concrete có thân hàm
    }
}

Ràng buộc biên dịch:

  • Không thể gọi: Animal a = new Animal(); (Báo lỗi biên dịch)
  • Vẫn chứa Constructor bình thường để lớp con kế thừa gọi qua super()
  • Chứa đầy đủ các thuộc tính và phương thức như lớp thông thường
Ví dụ mã nguồn đơn giản
abstract class Animal {
    abstract void sound();
    void eat() {
        System.out.println("Eating...");
    }
}

class Dog extends Animal {
    @Override
    void sound() {
        System.out.println("Woof Woof");
    }
}
Ví dụ thực tế trong dự án

Thiết lập lớp thực thể cơ sở (Base Entity) lưu vết thời gian trong các bảng cơ sở dữ liệu:

public abstract class BaseEntity {
    protected LocalDateTime createdAt;
    protected LocalDateTime updatedAt;

    public void updateTimestamp() {
        this.updatedAt = LocalDateTime.now();
    }
}

public class User extends BaseEntity {
    // Tự động có các thuộc tính createdAt, updatedAt và hàm updateTimestamp
}

8.2 Interface

Khái niệm và mục đích sử dụng

Interface là bản hợp đồng (giao kèo) thiết lập các hành vi bắt buộc mà các lớp triển khai phải tuân thủ.

Mục đích:

  • Giải quyết bài toán đa kế thừa hành vi trong Java (một Class có thể thực thi nhiều Interface cùng lúc)
  • Giảm thiểu tối đa sự liên kết cứng (Loose Coupling) giữa các module nghiệp vụ
  • Nền tảng hỗ trợ cơ chế Dependency Injection của Spring Boot
Cơ chế hoạt động bên trong

Kiến trúc thiết kế kết nối lỏng lẻo:

PaymentService (Interface) ├── PaypalServiceImpl (Implementation) ├── StripeServiceImpl (Implementation) └── MomoServiceImpl (Implementation)

JVM sẽ tự động tìm kiếm và thực thi phương thức của lớp triển khai cụ thể được chọn chạy ở thời điểm Runtime.

Cú pháp và quy tắc
public interface Flyable {
    void fly(); // Mặc định là public abstract method
}

public class Bird implements Flyable {
    @Override
    public void fly() {
        System.out.println("Bird is flying...");
    }
}
Ví dụ mã nguồn đơn giản
interface Payment {
    void pay();
}

class MomoPayment implements Payment {
    @Override
    public void pay() {
        System.out.println("Thanh toán qua ví MoMo");
    }
}
Ví dụ thực tế trong dự án

Khai báo Interface Service phân tách lớp nghiệp vụ Service trong dự án Spring Boot:

public interface UserService {
    User findById(Long id);
}

@Service
public class UserServiceImpl implements UserService {
    @Override
    public User findById(Long id) {
        return userRepository.findById(id).orElseThrow();
    }
}

8.3 Enum (Kiểu dữ liệu liệt kê)

Khái niệm và mục đích sử dụng

Enum (Enumeration) là một kiểu dữ liệu đặc biệt dùng để biểu diễn một danh sách cố định hữu hạn các giá trị hằng số không đổi.

Mục đích:

  • Loại bỏ việc sử dụng các chuỗi String hard-code dễ gõ sai chính tả
  • Đảm bảo tính an toàn kiểu dữ liệu cao (Typesafety)
  • Giúp mã nguồn rõ ràng, tự nhiên
Cơ chế hoạt động bên trong

Thực chất, mỗi hằng số trong Enum được JVM biên dịch thành một thực thể đối tượng hằng (static final object) được khởi sinh duy nhất một lần khi ứng dụng khởi chạy.

Mã nguồn khi bạn viết:

enum Role { ADMIN, STAFF }

Được JVM hiểu ngầm định tương đương với cấu trúc class:

public final class Role extends java.lang.Enum<Role> {
    public static final Role ADMIN = new Role("ADMIN", 0);
    public static final Role STAFF = new Role("STAFF", 1);
}
Cú pháp và quy tắc
enum Status {
    PENDING,
    APPROVED,
    REJECTED
}
Ví dụ mã nguồn đơn giản
enum Gender {
    MALE, FEMALE
}

public class Main {
    public static void main(String[] args) {
        Gender gender = Gender.MALE;
        System.out.println(gender); // MALE
    }
}
Ví dụ thực tế trong dự án

Kiểm tra quyền truy cập hệ thống của người dùng an toàn dữ liệu:

public enum UserRole {
    ADMIN, STAFF, CUSTOMER
}

// Xử lý logic check:
if (user.getRole() == UserRole.ADMIN) {
    System.out.println("Cho phép truy cập trang quản trị Admin");
}

8.4 Inner Class (Lớp lồng nhau)

Khái niệm và mục đích sử dụng

Inner Class là một lớp được khai báo nằm bên trong phạm vi dấu ngoặc nhọn của một lớp khác (Outer Class).

Mục đích:

  • Gom nhóm logic các class có mối liên hệ mật thiết khép kín vào cùng 1 nơi
  • Tăng tính đóng gói bảo mật (lớp ngoài có thể ẩn lớp trong hoàn toàn bằng private)
  • Giúp mã nguồn gọn gàng, giảm số lượng file class phụ
Cơ chế hoạt động bên trong

Một Inner Class thông thường (phi tĩnh) sở hữu một tham chiếu ngầm định trỏ ngược về thực thể của Outer Class chứa nó. Nhờ vậy, Inner Class có toàn quyền truy xuất trực tiếp tới mọi thuộc tính và phương thức private của Outer Class.

Cú pháp và quy tắc
class Car {
    private String brand = "Toyota";

    class Engine {
        void start() {
            System.out.println(brand + " Engine is starting..."); // Gọi trực tiếp private brand
        }
    }
}
Ví dụ mã nguồn đơn giản
public class Car {
    private String brand;
    public Car(String brand) { this.brand = brand; }

    class Engine {
        public void start() {
            System.out.println(brand + " engine started!");
        }
    }
}

// Gọi khởi sinh:
Car car = new Car("Honda");
Car.Engine engine = car.new Engine(); // Khởi tạo thông qua thực thể car
engine.start();
Ví dụ thực tế trong dự án

Cấu trúc giao diện Map.Entry<K, V> dùng để duyệt Map thực chất là một Inner Interface nằm bên trong lớp cha java.util.Map.

8.5 Anonymous Class (Lớp vô danh)

Khái niệm và mục đích sử dụng

Anonymous Class là lớp không có tên định danh cụ thể, được khai báo và khởi tạo thực thể trực tiếp cùng lúc ngay tại câu lệnh sử dụng.

Mục đích: Viết nhanh mã triển khai của một Interface hoặc ghi đè lớp khác chỉ để sử dụng duy nhất một lần, giúp tránh tạo file class vật lý phụ rác dự án.

Cơ chế hoạt động bên trong

Khi biên dịch mã nguồn, trình biên dịch Java sẽ tự động tạo ra một file class bytecode riêng biệt cho lớp vô danh này với cách đặt tên tự sinh theo định dạng:

OuterClassName$1.class
OuterClassName$2.class
Cú pháp và quy tắc
InterfaceName object = new InterfaceName() {
    @Override
    public void method() {
        // Viết code thân triển khai ở đây
    }
};
Ví dụ mã nguồn đơn giản
interface Greeting {
    void sayHello();
}

public class Main {
    public static void main(String[] args) {
        Greeting greeting = new Greeting() {
            @Override
            public void sayHello() {
                System.out.println("Hello Java Anonymous Class");
            }
        };
        greeting.sayHello();
    }
}
Ví dụ thực tế trong dự án

Trước khi Java 8 ra đời, Anonymous Class cực kỳ phổ biến khi lắng nghe sự kiện nút bấm Click hoặc tạo luồng chạy song song:

button.addActionListener(new ActionListener() {
    @Override
    public void actionPerformed(ActionEvent e) {
        System.out.println("Nút đã được click!");
    }
});

Ngày nay, hầu hết các biểu thức Anonymous Class này đã được viết tối giản thay thế bằng cú pháp Lambda Expression ngắn gọn:

button.addActionListener(e -> System.out.println("Nút đã được click!"));

So sánh Abstract Class và Interface

Tiêu chí so sánh Abstract Class (Lớp trừu tượng) Interface (Giao diện)
Hàm khởi tạo (Constructor) Có thể chứa Constructor để lớp con gọi qua super(). Tuyệt đối không chứa Constructor.
Thuộc tính (Variables) Có thể chứa các thuộc tính thông thường với mọi access modifier. Chỉ chứa hằng số mặc định là public static final.
Kế thừa (Inheritance) Đơn kế thừa (lớp con chỉ extends duy nhất 1 lớp cha). Đa thực thi (lớp con có thể implements nhiều interface).
Phương thức (Methods) Chứa được cả phương thức abstract rỗng và concrete có code thân. Mặc định là abstract method. Chỉ chứa code thân với hàm default/static (từ Java 8).
Mục đích chính Thiết lập mối quan hệ gia đình hệ thống ("is-a"), chia sẻ code dùng chung. Thiết lập bản hợp đồng cam kết các hành vi hệ thống ("can-do").
Khi nào chọn sử dụng?
  • Chọn dùng Abstract Class: Khi các lớp có mối liên hệ gia đình rõ ràng (ví dụ: Chó, Mèo, Chim đều Kế thừa từ lớp Động Vật Animal), có nhu cầu dùng chung mã nguồn và cần lưu trữ các thuộc tính biến đổi trạng thái (state).
  • Chọn dùng Interface: Khi bạn chỉ quan tâm đến việc quy chuẩn hành vi chung cho các lớp không có liên hệ gia đình (ví dụ: Hóa Đơn và Xe Hơi đều có khả năng Printable - in ấn), hoặc khi thiết kế kiến trúc các module nghiệp vụ kết nối lỏng lẻo (như Service, Repository).
