10.1 Exception là gì

Khái niệm và mục đích sử dụng

Exception (Ngoại lệ) là một sự kiện bất thường xảy ra trong quá trình thực thi chương trình, làm gián đoạn hoặc phá hỏng luồng chạy tuần tự bình thường của mã nguồn.

Mục đích sử dụng:

  • Ngăn chặn hệ thống bị sập hay dừng chạy đột ngột khi gặp lỗi thô (Graceful Degradation)
  • Định nghĩa cơ chế ghi lại thông tin lỗi chi tiết giúp phục vụ việc gỡ lỗi (Debugging)
  • Tách rời mã nguồn xử lý nghiệp vụ chính và mã nguồn xử lý lỗi giúp hệ thống sạch hơn
Cơ chế hoạt động bên trong (Lan truyền ngoại lệ - Exception Propagation)

Khi một lỗi xảy ra trong phương thức, JVM tạo ra một đối tượng Exception chứa thông tin lỗi và bắt đầu tìm kiếm khối xử lý:

Phương thức hiện tại phát sinh lỗi ↓ JVM tìm kiếm khối catch tại phương thức này ├── Có catch khớp → Nhảy vào xử lý lỗi └── Không có catch ↓ (Lan truyền ngược về nơi gọi hàm - Propagation) Tiếp tục đẩy Exception lên phương thức cha gọi nó ... Không có bất kỳ nơi nào bắt catch ↓ JVM dừng luồng chạy đột ngột, in ra Stack Trace lỗi.
Cú pháp và quy tắc

Ví dụ một lỗi chia số học cho 0:

int result = 10 / 0; // JVM lập tức ném ra ArithmeticException
Ví dụ mã nguồn đơn giản
public class Main {
    public static void main(String[] args) {
        int a = 10;
        int b = 0;
        int result = a / b; // Chương trình sập tại đây
        System.out.println(result);
    }
}

Màn hình console báo lỗi sập:

Exception in thread "main" java.lang.ArithmeticException: / by zero
Ví dụ thực tế trong dự án

Chủ động ném ngoại lệ khi không tìm thấy dữ liệu người dùng yêu cầu:

User user = userRepository.findById(id)
        .orElseThrow(() -> new RuntimeException("Không tìm thấy người dùng này!"));

10.2 Checked Exception

Khái niệm và mục đích sử dụng

Checked Exception là những ngoại lệ phát sinh ngoài tầm kiểm soát của ứng dụng và **bắt buộc phải được kiểm tra (xử lý)** ngay tại thời điểm biên dịch code (Compile Time).

Các Checked Exception phổ biến:

  • IOException (Lỗi vào ra dữ liệu luồng)
  • SQLException (Lỗi truy vấn cơ sở dữ liệu)
  • FileNotFoundException (Không tìm thấy tệp tin vật lý)
  • ParseException (Lỗi chuyển đổi kiểu dữ liệu ngày tháng)
Cơ chế hoạt động bên trong

Nếu bạn viết một câu lệnh đọc file thô:

FileReader reader = new FileReader("data.txt");

Trình biên dịch Java phát hiện dòng code này có nguy cơ lỗi hệ thống tệp tin bên ngoài nên lập tức báo lỗi đỏ, bắt buộc bạn phải viết bao bọc nó trong try-catch hoặc khai báo throws ở signature của hàm mới cho phép build dự án.

Cú pháp và quy tắc

Có 2 cách giải quyết bắt buộc:

Cách 1: Bao bọc xử lý trực tiếp bằng try-catch:

try {
    FileReader reader = new FileReader("data.txt");
} catch (FileNotFoundException e) {
    System.out.println("Hãy kiểm tra lại đường dẫn file");
}

Cách 2: Khai báo né tránh lỗi thông qua từ khóa throws ở chữ ký phương thức để đẩy trách nhiệm cho nơi gọi nó xử lý:

public void readFile() throws FileNotFoundException {
    FileReader reader = new FileReader("data.txt");
}
Ví dụ thực tế trong dự án

Khai báo API xuất dữ liệu báo cáo sang file Excel có rủi ro luồng ghi:

public void exportExcelReport() throws IOException {
    // Code xử lý ghi workbook xuất file
}

10.3 Unchecked Exception (Runtime Exception)

Khái niệm và mục đích sử dụng

Unchecked Exception là những ngoại lệ phát sinh trong suốt quá trình chạy chương trình (Runtime), thường xảy ra do lỗi logic trong code của lập trình viên.

Trình biên dịch **không bắt buộc** bạn phải khai báo xử lý lỗi Unchecked bằng try-catch khi viết code.

Các Unchecked Exception phổ biến:

  • NullPointerException (Lỗi tham chiếu con trỏ null)
  • ArithmeticException (Lỗi số học như chia cho 0)
  • ArrayIndexOutOfBoundsException (Lỗi chỉ số index mảng nằm ngoài phạm vi)
  • IllegalArgumentException (Truyền đối số đầu vào không hợp lệ)
  • NumberFormatException (Ép chuỗi không phải chữ số sang số nguyên)
Cơ chế hoạt động bên trong

Tất cả Unchecked Exception đều kế thừa từ lớp cha java.lang.RuntimeException. Lỗi chỉ được phát hiện khi luồng dữ liệu lỗi chạy qua dòng lệnh thực tế lúc vận hành hệ thống.

Ví dụ mã nguồn đơn giản
public class Main {
    public static void main(String[] args) {
        String data = "abc";
        int number = Integer.parseInt(data); // Ném ra NumberFormatException tại Runtime
    }
}
Ví dụ thực tế trong dự án

Chủ động ném ngoại lệ Runtime khi tham số truyền vào từ client vi phạm nguyên tắc nghiệp vụ:

if (orderQuantity <= 0) {
    throw new IllegalArgumentException("Số lượng mua sản phẩm bắt buộc phải lớn hơn 0");
}

10.4 try - catch - finally

Khái niệm và mục đích sử dụng

Đây là khối xử lý kiểm soát ngoại lệ cơ bản nhất:

  • try: Chứa khối câu lệnh có rủi ro cao gây ra lỗi ngoại lệ.
  • catch: Nơi bắt và xử lý lỗi cụ thể nếu xảy ra ngoại lệ trong khối try.
  • finally: Khối mã nguồn **bắt buộc luôn luôn được thực thi** dù có ngoại lệ xảy ra hay không.
Cơ chế hoạt động bên trong

Dù trong khối try hoặc catch có chứa lệnh ngắt hàm return, JVM vẫn đảm bảo phải nhảy vào thực thi xong toàn bộ mã nguồn của khối finally trước khi thực hiện trả về kết quả cho hàm. Điều này cực kỳ quan trọng dùng để giải phóng tài nguyên.

Cú pháp và quy tắc
try {
    // Code có thể lỗi
} catch (SpecificException e) {
    // Xử lý lỗi cụ thể
} catch (Exception e) {
    // Bắt các lỗi chung chung còn lại (luôn đặt Exception cha ở catch cuối cùng)
} finally {
    // Luôn giải phóng tài nguyên tại đây
}

Cơ chế tự đóng tài nguyên (Try-with-resources): Từ Java 7+, với các tài nguyên implement giao diện AutoCloseable (như Connection, InputStream), ta nên khai báo trực tiếp trong ngoặc đơn sau try để JVM tự động đóng khi kết thúc, không cần viết giải phóng ở finally thủ công nữa:

try (Connection conn = dataSource.getConnection()) {
    // Thực thi database... (Tự động đóng conn khi hoàn thành)
}
Ví dụ mã nguồn đơn giản
public class Main {
    public static void main(String[] args) {
        try {
            int val = 10 / 0;
        } catch (ArithmeticException e) {
            System.out.println("Không thể thực hiện chia cho 0");
        } finally {
            System.out.println("Luôn in ra câu này");
        }
    }
}

10.5 throw và throws

Khái niệm và mục đích sử dụng
  • throw: Dùng để chủ động bắn (ném) ra một đối tượng Exception cụ thể tại một dòng code.
  • throws: Khai báo ở chữ ký phương thức cảnh báo nơi gọi hàm rằng phương thức này có khả năng ném ra các ngoại lệ Checked.
Cú pháp và quy tắc

Sử dụng throw:

throw new RuntimeException("Thông điệp lỗi");

Sử dụng throws:

public void checkDatabase() throws SQLException, IOException { ... }
Ví dụ thực tế trong dự án

Chủ động kiểm tra và ném lỗi nghiệp vụ trong API Spring Boot:

if (user == null) {
    throw new RuntimeException("Tài khoản không tồn tại trên hệ thống!");
}

// Đối với Spring Boot REST API:
throw new ResponseStatusException(HttpStatus.NOT_FOUND, "Không tìm thấy người dùng");

10.6 Custom Exception (Tự định nghĩa ngoại lệ)

Khái niệm và mục đích sử dụng

Custom Exception là ngoại lệ do lập trình viên tự tạo ra bằng cách kế thừa từ các lớp Exception của Java nhằm mô tả chính xác và phân loại các nghiệp vụ đặc thù của doanh nghiệp.

Ví dụ thực tế:

  • UserNotFoundException (Không tìm thấy tài khoản)
  • InsufficientBalanceException (Số dư tài khoản không đủ thanh toán)
  • InvalidPasswordException (Mật khẩu nhập sai)
Cơ chế hoạt động bên trong

Trong các dự án Spring Boot hiện đại, đa số Custom Exception được kế thừa trực tiếp từ RuntimeException (Unchecked) để tránh làm rối code ở tầng Controller bởi các khai báo throws.

Cú pháp và quy tắc
public class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(String message) {
        super(message); // Đẩy thông điệp lỗi lên RuntimeException xử lý
    }
}
Ví dụ thực tế trong dự án

1. Định nghĩa Custom Exception:

public class UserNotFoundException extends RuntimeException {
    public UserNotFoundException(Long id) {
        super("Không tìm thấy người dùng có mã ID: " + id);
    }
}

2. Gọi ném lỗi ở tầng Service:

User user = userRepository.findById(id)
        .orElseThrow(() -> new UserNotFoundException(id));

3. Kết hợp bắt lỗi tập trung (Global Exception Handler) bằng @RestControllerAdvice ở Spring Boot:

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<String> handleUserNotFound(UserNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(ex.getMessage());
    }
}

Nhờ cấu hình Global Handler này, API trả về mã HTTP 404 chuẩn kèm thông điệp lỗi tự tạo cực kỳ đẹp đẽ và sạch sẽ.

Cây phân cấp Exception trong Java

Throwable / \ (Lỗi hệ thống) / \ (Ngoại lệ ứng dụng) v v Error Exception / \ (Checked) / \ (Unchecked) v v IOException RuntimeException ... / \ v v NullPointerException ArithmeticException ...
Quyết định lựa chọn sử dụng
  • Sử dụng Checked Exception khi: Gặp lỗi khách quan, hệ thống có khả năng tự phục hồi xử lý lại sau khi bắt lỗi (Ví dụ: File bị mất thì báo người dùng chọn file khác, kết nối DB lỗi thì thử kết nối lại).
  • Sử dụng Unchecked Exception khi: Gặp các lỗi logic kỹ thuật lập trình của developer (Ví dụ: truyền giá trị null, chỉ số mảng vượt biên, ép sai kiểu chữ sang số) không thể phục hồi giữa chừng, bắt buộc phải sửa code hoặc ném lỗi báo hiệu nghiệp vụ.
