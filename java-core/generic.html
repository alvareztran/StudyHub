12.1 Generic Class (Lớp khuôn mẫu dữ liệu)

Khái niệm và mục đích sử dụng

Generics (Tham số hóa kiểu dữ liệu) là cơ chế cho phép định nghĩa lớp, phương thức, giao diện với các tham số kiểu thay thế linh hoạt được kiểm soát chặt chẽ an toàn kiểu dữ liệu tại thời điểm biên dịch (Compile Time).

Sự cải tiến vượt bậc so với thời kỳ chưa có Generics (Java 5 về trước):

  • Trước Java 5 (Không có Generic): Mọi Collection lưu trữ dữ liệu dạng Object. Khi đọc dữ liệu, bắt buộc phải ép kiểu thủ công bằng tay (Casting), có nguy cơ cao xảy ra lỗi runtime sập ứng dụng ClassCastException.
  • Sau Java 5 (Có Generic): Kiểu dữ liệu được chỉ định cụ thể khi tạo đối tượng. JVM kiểm soát chặt chẽ kiểu truyền vào, không cần ép kiểu thủ công.
Cơ chế hoạt động bên trong (Xóa kiểu - Type Erasure)

Để đảm bảo tính tương thích ngược với các phiên bản Java cũ, Java sử dụng cơ chế **Type Erasure**. Khi mã nguồn được compile sang Bytecode:

Khi viết code: Box stringBox = new Box<>(); Box intBox = new Box<>(); ↓ (Trình biên dịch Compile) Xóa bỏ thông tin Generic, chuyển toàn bộ ký hiệu kiểu về lớp Object thô: Box stringBox = new Box(); Box intBox = new Box(); (Bên trong class Box, thuộc tính T value; chuyển thành Object value;)

Thông tin Generic chỉ tồn tại để compiler bắt lỗi lập trình viên, hoàn toàn không tồn tại khi chương trình chạy ở Runtime.

Cú pháp và quy tắc đặt tên hằng kiểu
class ClassName<T> {
    private T data;
}

Các ký hiệu quy chuẩn phổ biến đặt tên tham số Generic:

Ký hiệu Mô tả quy ước
TType (Kiểu dữ liệu chung)
EElement (Phần tử trong Collection như List)
KKey (Khóa dùng trong Map)
VValue (Giá trị ánh xạ trong Map)
NNumber (Chỉ nhận các kiểu số)
Ví dụ mã nguồn đơn giản
class Box<T> {
    private T value;

    public void setValue(T value) { this.value = value; }
    public T getValue() { return this.value; }
}

public class Main {
    public static void main(String[] args) {
        Box<String> box1 = new Box<>();
        box1.setValue("Java"); // Chỉ nhận String
        System.out.println(box1.getValue());
    }
}
Ví dụ thực tế trong dự án

Thiết lập khuôn mẫu phản hồi API thống nhất (Wrapper API Response) cho dự án Microservice:

public class ApiResponse<T> {
    private boolean success;
    private T data; // Chứa Object dữ liệu động tùy thuộc thực thể gọi
}

// Sử dụng thực tế:
ApiResponse<User> userResponse;
ApiResponse<List<Product>> productsListResponse;

12.2 Generic Method

Khái niệm và mục đích sử dụng

Generic Method là phương thức có khả năng xử lý nhiều kiểu dữ liệu tham số đầu vào độc lập, không phụ thuộc vào việc Class chứa nó có phải là Generic Class hay không.

Cú pháp và quy tắc
public <T> returnType methodName(T parameter) { ... }

Quy tắc: Khai báo tham số Generic <T> bắt buộc phải được đặt trước kiểu dữ liệu trả về của phương thức.

Ví dụ mã nguồn đơn giản
public class Printer {
    // Phương thức tĩnh Generic
    public static <T> void print(T value) {
        System.out.println(value);
    }

    public static void main(String[] args) {
        print("Hello"); // Compiler tự động suy luận T = String
        print(100);     // Compiler tự động suy luận T = Integer
    }
}

12.3 Wildcard (Ký hiệu wildcard ?)

Khái niệm và mục đích sử dụng

Wildcard biểu diễn bằng ký tự dấu hỏi chấm ?. Dùng làm đại diện cho một kiểu dữ liệu chưa xác định rõ ràng tại signature của Generic Collection, giúp tăng tối đa tính linh hoạt cho API.

Phân loại các giới hạn Wildcard (Bounds)

1. Unbounded Wildcard (Không giới hạn)

Chấp nhận truyền vào mọi kiểu dữ liệu của Collection:

List<?> list; // Chấp nhận List<String>, List<Integer>...

2. Upper Bounded Wildcard (Giới hạn cận trên)

Chỉ chấp nhận các kiểu dữ liệu kế thừa từ lớp cha được chỉ định cụ thể:

List<? extends Number> list; // Chấp nhận List<Integer>, List<Double>... (Không nhận String)

3. Lower Bounded Wildcard (Giới hạn cận dưới)

Chỉ chấp nhận các kiểu dữ liệu là cha trực tiếp hoặc gián tiếp của lớp được chỉ định:

List<? super Integer> list; // Chấp nhận List<Integer>, List<Number>, List<Object>
Ví dụ mã nguồn đơn giản
public class Calculator {
    // Tính tổng mọi loại List số học (Integer, Double...)
    public static double sum(List<? extends Number> list) {
        double total = 0.0;
        for (Number number : list) {
            total += number.doubleValue();
        }
        return total;
    }
}

Quy tắc PECS (Producer Extends, Consumer Super)

Quy tắc PECS là nguyên lý thiết kế Generics tối thượng trong Java khi thao tác với mảng/danh sách Wildcard:

PECS Rule: ├── Producer Extends () -> Chỉ đọc dữ liệu ra khỏi danh sách └── Consumer Super () -> Chỉ ghi (thêm) dữ liệu vào danh sách
  • Producer Extends: Nếu phương thức của bạn chỉ làm nhiệm vụ lấy (đọc) phần tử ra khỏi Collection để tính toán, hãy dùng <? extends T>. JVM không cho phép bạn ghi dữ liệu vào collection này để tránh lỗi sai kiểu.
  • Consumer Super: Nếu phương thức của bạn chỉ làm nhiệm vụ ghi thêm (tiêu thụ) phần tử vào Collection, hãy dùng <? super T>.
Ứng dụng PECS trong bộ SDK Java core

Hãy xem phương thức sao chép danh sách của lớp tiện ích java.util.Collections:

public static <T> void copy(List<? super T> dest, List<? extends T> src) {
    for (int i = 0; i < src.size(); i++) {
        dest.set(i, src.get(i)); // src là Producer (đọc ra), dest là Consumer (ghi vào)
    }
}
